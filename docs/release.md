# Release process

This page documents how to cut a `zunel` release that simultaneously
publishes to GitHub Releases, the personal Homebrew tap, and as `.deb`
artifacts for Debian/Ubuntu. It also covers the one-time setup steps
that live outside this repo (creating the tap repo, minting the
Homebrew PAT) and the future GH-Pages apt repo work.

## Overview

There are two CI workflows:

- `.github/workflows/release.yml` — generated by [`cargo-dist`][dist].
  Triggered on tags matching `**[0-9]+.[0-9]+.[0-9]+*` (so `v0.2.0`,
  `0.2.0`, `app/v1.0.0-rc.1`, etc. all match). Builds per-target binary
  tarballs, opens a GitHub Release with auto-generated notes, then
  pushes a Homebrew formula to the configured tap.
- `.github/workflows/deb.yml` — fires on `workflow_run: ["Release"] /
  completed`, downloads the per-arch musl tarballs the upstream run
  attached to the GitHub Release, runs `cargo-deb` against the
  already-stripped binaries, and re-uploads the resulting `.deb`
  files to the same release. (We listen on `workflow_run` instead of
  `release: published` because GitHub Actions intentionally suppresses
  `release.published` when the release is created by a workflow using
  `GITHUB_TOKEN` — which `Release > host` does. The job has a
  conclusion-success guard so it's a no-op on failed Release runs.)

Cargo-dist's per-platform targets (configured in
`rust/Cargo.toml → [workspace.metadata.dist].targets`) are:

| Triple                            | Audience                   |
| --------------------------------- | -------------------------- |
| `aarch64-apple-darwin`            | macOS Apple Silicon        |
| `x86_64-apple-darwin`             | macOS Intel                |
| `aarch64-unknown-linux-musl`      | Linux arm64 (static musl)  |
| `x86_64-unknown-linux-musl`       | Linux amd64 (static musl)  |

Both Linux targets feed cargo-dist's tarball publish **and** the
deb-build workflow. macOS targets feed the tarball publish + the
Homebrew formula.

## One-time setup (do once per checkout-of-record)

These steps live outside this repo and have to be done by the
maintainer with a GitHub session.

1. **Create the personal Homebrew tap repo.** On GitHub, create an
   empty public repo named **`rdu16625/homebrew-tap`** (the
   `homebrew-` prefix is mandatory — Homebrew matches it when a user
   runs `brew tap rdu16625/tap`). No README needed; cargo-dist will
   commit `Formula/zunel.rb` on the first release. The tap name is
   already pinned in `rust/Cargo.toml → [workspace.metadata.dist].tap`.

2. **Mint a fine-grained PAT scoped to the tap repo.** GitHub →
   Settings → Developer settings → Personal access tokens → Fine-
   grained tokens → Generate new token. Limit it to the
   `rdu16625/homebrew-tap` repository, grant **Contents: Read and
   write**, leave everything else alone.

3. **Add the PAT as a repo secret on `rdu16625/zunel`.** Settings →
   Secrets and variables → Actions → New repository secret →
   `HOMEBREW_TAP_TOKEN` = the PAT minted above. The release workflow
   references this secret name verbatim at `release.yml:297`.

4. **(Optional) Verify locally.** With `cargo-dist` 0.31 installed
   (already in `~/.cargo/bin/dist`):

   ```bash
   (cd rust && dist plan)
   ```

   Should report the same per-target tarballs the live release will
   produce, plus a `homebrew` installer artifact.

That's all the manual setup. Cutting subsequent releases is fully
automated.

## Cutting a release

```bash
# 1. Bump the workspace version. Single line in rust/Cargo.toml:
#      version = "0.2.0"  →  version = "0.3.0"
$EDITOR rust/Cargo.toml

# 2. Commit, run the local CI gate, push.
git commit -am "release: 0.3.0"
scripts/check.sh
git push

# 3. Tag and push the tag. The tag triggers release.yml.
git tag v0.3.0
git push origin v0.3.0
```

GitHub Actions then:

1. `release.yml > plan/build-local-artifacts/build-global-artifacts/host`
   builds the four per-target tarballs in parallel, computes SHA256s,
   creates a GitHub Release, uploads the tarballs.
2. `release.yml > publish-homebrew-formula` clones
   `rdu16625/homebrew-tap` (using `HOMEBREW_TAP_TOKEN`), drops a fresh
   `Formula/zunel.rb` referencing the new tarballs, commits, pushes.
3. `release.yml > announce` finalizes the GitHub Release.
4. The release-published event triggers `deb.yml`, which builds and
   attaches `zunel-amd64.deb` and `zunel-arm64.deb`.

Total wall-clock is ~10 minutes end-to-end on cold caches.

After the release lands, users see:

```bash
brew upgrade zunel               # macOS
sudo dpkg -i ./zunel-amd64.deb   # Debian/Ubuntu
```

## Editing the release pipeline

`release.yml` is **regenerated** from `[workspace.metadata.dist]` by
running `(cd rust && dist generate --mode=ci)`. The repo keeps
`allow-dirty = ["ci"]` set so cargo-dist refuses to overwrite the
hand-patched workflow without explicit consent — the patches are
necessary because cargo-dist assumes the Cargo workspace lives at the
repository root, but ours is under `rust/`.

When changing dist metadata (new platform target, new installer,
upgrading `cargo-dist-version`, switching tap, etc.):

```bash
# 1. Edit [workspace.metadata.dist] in rust/Cargo.toml.

# 2. Snapshot the current hand-patched workflow.
cp .github/workflows/release.yml /tmp/release.yml.handpatched

# 3. Temporarily allow dist to overwrite, regenerate, restore the
#    safety flag.
sed -i.bak '/^allow-dirty = \["ci"\]$/d' rust/Cargo.toml
(cd rust && dist generate --mode=ci)
git checkout rust/Cargo.toml.bak 2>/dev/null || true
# Re-add the line by hand; sed deleted it.
$EDITOR rust/Cargo.toml

# 4. Re-apply the workspace-in-subdir patches.
diff -U2 /tmp/release.yml.handpatched .github/workflows/release.yml
# For each `dist <subcmd>` step in the regenerated file, add
# `working-directory: rust`. The exact set of patches lives in the
# diff above the new `publish-homebrew-formula` job.
#
# IMPORTANT: in the post-build steps that compute the upload `paths`
# output, use `dist print-upload-files-from-manifest --manifest
# dist-manifest.json` *as-is* (no `sed 's#^#rust/#'` prefix). cargo-dist
# already prints absolute paths under `/home/runner/.../rust/target/
# distrib/...`, so prefixing them with `rust/` produces malformed
# `rust//home/...` paths and actions/upload-artifact silently drops
# the tarballs. Same for the global-artifacts job; replace any
# `jq --raw-output .upload_files[]` invocation with the canonical
# `dist print-upload-files-from-manifest` command.

# 5. Re-apply the formula-bump attribution patch in
#    `publish-homebrew-formula.env`:
#       GITHUB_USER: "github-actions[bot]"
#       GITHUB_EMAIL: "41898282+github-actions[bot]@users.noreply.github.com"
#    (cargo-dist regenerates these to its own `axo bot` defaults; the
#    homebrew-tap repo's history stays consistent if every commit lands
#    under the standard `github-actions[bot]` identity.)

# 6. Re-apply the `Sync Rust toolchain to project pin` step in the
#    `build-local-artifacts` job, immediately before `Install dist`:
#       - name: Sync Rust toolchain to project pin
#         working-directory: rust
#         shell: bash
#         run: |
#           rustup show active-toolchain || rustup show
#    Required because cargo-dist's generated workflow does not by
#    default auto-install the toolchain pinned in rust-toolchain.toml,
#    and GitHub-hosted Linux runners ship with a cached rustc that is
#    often below the workspace MSRV.

# 7. Verify and ship.
scripts/check.sh
```

## Editing the .deb pipeline

`deb.yml` is hand-written (cargo-dist 0.31 doesn't ship a `.deb`
installer). To adjust packaging metadata edit
`[package.metadata.deb]` in `rust/crates/zunel-cli/Cargo.toml`. To
smoke-test a build locally on macOS or Linux:

```bash
cargo install cargo-deb --locked
cd rust && cargo build --release -p zunel-cli && cd ..
(cd rust/crates/zunel-cli && \
   cargo deb --no-build --no-strip --output ../../target/zunel-host.deb)
ar t rust/target/zunel-host.deb           # debian-binary, control, data
```

(The `dpkg-shlibdeps failed to launch` warning on macOS is benign;
shlibdeps only runs in real Linux build environments.)

## Future apt repo on GH-Pages

Today, "install via apt" requires the user to download the `.deb` and
run `dpkg -i`. To get the user-friendly `apt install zunel` UX, the
remaining work is:

1. Mint a GPG signing key for the apt repo. Store the private key as a
   GitHub Actions secret (`APT_REPO_SIGNING_KEY`).
2. Reserve the GH-Pages branch on `rdu16625/zunel` (or a separate
   `rdu16625/apt` repo) for the static apt index.
3. Extend `deb.yml` with a final job that, after both arch builds
   succeed, clones the apt-repo branch, drops the new `.deb`s into
   `pool/main/z/zunel/`, regenerates `Packages.gz` / `Release`
   indexes via `apt-ftparchive`, signs `Release` with the imported
   GPG key, and pushes.
4. Document the user-side setup in `docs/quick-start.md`:

   ```bash
   curl -fsSL https://rdu16625.github.io/apt/zunel-archive-keyring.gpg \
     | sudo gpg --dearmor -o /etc/apt/keyrings/zunel.gpg
   echo "deb [signed-by=/etc/apt/keyrings/zunel.gpg] https://rdu16625.github.io/apt stable main" \
     | sudo tee /etc/apt/sources.list.d/zunel.list
   sudo apt update && sudo apt install zunel
   ```

Until that lands, the dpkg-direct path documented in
`docs/quick-start.md` is the supported install flow.
